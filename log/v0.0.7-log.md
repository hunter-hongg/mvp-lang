# 🦉 Miva v0.0.7 Released
>
> **Silent Owl**

We are pleased to announce the official release of **Miva Programming Language v0.0.7**!
This version focuses on stabilizing type systems, introducing compiler directives, establishing error & warning codes, enabling engineering configurations, and standardizing core syntax. Miva officially progresses from **functional** to **usable, robust, and maintainable**.

---

## 🔧 Core Updates

### 1. Full Fixes for Type Inference

- Added a built-in function type registry to uniformly supplement return and parameter types for all built-in interfaces.
- Fixed missing type information for loop variables in `for` loops. Iteration variables are now injected into the environment scope, and behave identically to regular local variables.
- Unified strict type checking across all functions, including `unsafe` functions.

### 2. New Magic Comment (Compiler Directives) System

A dedicated `/!` directive syntax is introduced to control compiler behavior with fine-grained precision.

- `/!warning_off <code>`: Disable specified warning codes
- `/!warning_err <code>`: Promote specified warnings to hard errors that abort compilation
- `/!release <mode>`: Configure optimization levels, overriding global CLI arguments
- `/!mangle <on|off>`: Control symbol mangling, designed specifically for FFI interoperability to generate C ABI-compatible symbols.

All directives follow strict leading-line syntax rules, with scope limited to the enclosing block.

### 3. Semantic Annotation System

We clearly distinguish **compiler directives** from **semantic code annotations**. The new `@` annotation system does not affect compilation or intermediate representation, and serves code documentation, IDE tooling, and code review.

- `@impl`: Describes traits/interfaces implemented by a structure
- `@unsafe`: Explains potential risks and preconditions of unsafe functions
- `@trusted`: Justifies why an unsafe call remains memory-safe within a trusted function.

### 4. Standardized Error & Warning Code Scheme

A complete, unique layered coding system is established for all compile-time diagnostics:

- Error codes: `E0001 ~ E0999` (hard compilation failures)
- Warning codes: `W0001 ~ W0999` (code style, deprecation, and semantic risks)

Every diagnostic is assigned a unique ID, enabling precise warning control and toolchain integration.

### 5. Mandatory Semicolon Syntax Introduced

Prior to v0.0.7, Miva had **no semicolon requirement at all**. This release formally enforces explicit semicolons to eliminate parser ambiguity and improve compiler stability.

**Semicolons are required after:**

- All statements inside function bodies
- All standalone expressions inside function bodies
- Module declarations
- Module imports
- Function exports

### 6. Multi-Level Compiler Configuration System

A layered configuration mechanism is implemented with clear priority rules:

- Global user config: `~/.miver/config/miva.toml`
- Local project config: `./.miva.toml`

Priority order (lowest to highest): Built-in defaults → Global config → Local project config.
Configuration files use the TOML format.

---

## 🚫 Breaking Changes

### Mandatory Semicolon Requirement (Major Breaking Change)

All legacy Miva code written for prior versions **cannot compile on v0.0.7 without semicolon additions**.

Semicolons must be appended to:

- Module declarations
- Module imports
- Function exports
- All statements within function bodies
- All standalone expressions within function bodies
